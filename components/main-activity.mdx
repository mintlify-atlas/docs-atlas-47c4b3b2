## Overview

`MainActivity` serves as the primary entry point for the VetyCare application. It extends `AppCompatActivity` and is responsible for:

- Setting up the application toolbar and navigation
- Hosting the navigation graph with fragments
- Managing the Floating Action Button (FAB)
- Handling options menu interactions

The class is located at `app/src/main/java/com/example/vetycare/MainActivity.kt`.

## Lifecycle Methods

### onCreate()

The `onCreate()` method initializes the activity and sets up all core components:

<Steps>
  <Step title="Initialize View Binding">
    Inflate the activity layout using View Binding to get type-safe view references:
    
    ```kotlin
    binding = ActivityMainBinding.inflate(layoutInflater)
    setContentView(binding.root)
    ```
    
    This replaces the traditional `setContentView(R.layout.activity_main)` approach and provides compile-time safety.
  </Step>

  <Step title="Setup Toolbar">
    Configure the Material Toolbar as the activity's action bar:
    
    ```kotlin
    setSupportActionBar(binding.toolbar)
    ```
    
    The toolbar is defined in the layout with ID `toolbar` and is accessed through the binding object.
  </Step>

  <Step title="Configure Navigation">
    Set up the Navigation Component to handle fragment navigation:
    
    ```kotlin
    val navController = findNavController(R.id.nav_host_fragment_content_main)
    appBarConfiguration = AppBarConfiguration(navController.graph)
    setupActionBarWithNavController(navController, appBarConfiguration)
    ```
    
    This connects the toolbar with the navigation system, enabling automatic Up button handling and title updates.
  </Step>

  <Step title="Setup Floating Action Button">
    Configure the FAB with a click listener that displays a Snackbar:
    
    ```kotlin
    binding.fab.setOnClickListener { view ->
        Snackbar.make(view, "Replace with your own action", Snackbar.LENGTH_LONG)
            .setAction("Action", null)
            .setAnchorView(R.id.fab).show()
    }
    ```
    
    The FAB is positioned at the bottom-right corner of the screen and can be customized for app-specific actions.
  </Step>
</Steps>

### Complete onCreate Implementation

```kotlin MainActivity.kt:19-36
override fun onCreate(savedInstanceState: Bundle?) {
    super.onCreate(savedInstanceState)

    binding = ActivityMainBinding.inflate(layoutInflater)
    setContentView(binding.root)

    setSupportActionBar(binding.toolbar)

    val navController = findNavController(R.id.nav_host_fragment_content_main)
    appBarConfiguration = AppBarConfiguration(navController.graph)
    setupActionBarWithNavController(navController, appBarConfiguration)

    binding.fab.setOnClickListener { view ->
        Snackbar.make(view, "Replace with your own action", Snackbar.LENGTH_LONG)
            .setAction("Action", null)
            .setAnchorView(R.id.fab).show()
    }
}
```

## Navigation Support

### Up Button Handling

The `onSupportNavigateUp()` method enables the Up button in the toolbar to navigate back through the app's navigation hierarchy:

```kotlin MainActivity.kt:54-58
override fun onSupportNavigateUp(): Boolean {
    val navController = findNavController(R.id.nav_host_fragment_content_main)
    return navController.navigateUp(appBarConfiguration)
            || super.onSupportNavigateUp()
}
```

<Tip>
  This method delegates navigation handling to the Navigation Component, which automatically manages the back stack based on the navigation graph.
</Tip>

## Best Practices Demonstrated

<Warning>
  **View Binding:** The activity uses View Binding instead of `findViewById()`, which provides:
  - Null safety (views are always non-null)
  - Type safety (no casting required)
  - Better performance (binding is created at compile time)
</Warning>

<Note>
  **Late Initialization:** Properties like `binding` and `appBarConfiguration` use `lateinit` because they're initialized in `onCreate()` rather than in the constructor.
</Note>

<Tip>
  **Single Activity Architecture:** This design follows Android's recommended Single Activity pattern, where one activity hosts multiple fragments for different screens.
</Tip>

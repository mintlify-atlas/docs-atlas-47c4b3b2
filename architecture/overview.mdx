VetyCare is built using modern Android development practices with a focus on simplicity, maintainability, and Material Design 3 principles. This page provides an overview of the architectural decisions and patterns used throughout the application.

## Architecture Pattern

VetyCare uses the **Single Activity Architecture** pattern with the Android Navigation Component. This architecture provides:

<CardGroup cols={2}>
  <Card title="Single Entry Point" icon="door-open">
    One `MainActivity` manages the entire app lifecycle and hosts all fragments
  </Card>
  <Card title="Fragment-Based UI" icon="puzzle-piece">
    Each screen is implemented as a Fragment for modular UI components
  </Card>
  <Card title="Centralized Navigation" icon="map">
    Navigation Component handles all screen transitions and back stack management
  </Card>
  <Card title="View Binding" icon="link">
    Type-safe view access without findViewById() calls
  </Card>
</CardGroup>

## Core Components

<Steps>
  <Step title="MainActivity">
    The single activity that hosts all fragments and manages the navigation controller.
    
    ```kotlin
    class MainActivity : AppCompatActivity() {
        private lateinit var appBarConfiguration: AppBarConfiguration
        private lateinit var binding: ActivityMainBinding
        
        override fun onCreate(savedInstanceState: Bundle?) {
            super.onCreate(savedInstanceState)
            binding = ActivityMainBinding.inflate(layoutInflater)
            setContentView(binding.root)
            
            val navController = findNavController(R.id.nav_host_fragment_content_main)
            appBarConfiguration = AppBarConfiguration(navController.graph)
            setupActionBarWithNavController(navController, appBarConfiguration)
        }
    }
    ```
  </Step>
  
  <Step title="Fragments">
    Each screen is implemented as a Fragment with its own lifecycle and view binding.
    
    ```kotlin
    class FirstFragment : Fragment() {
        private var _binding: FragmentFirstBinding? = null
        private val binding get() = _binding!!
        
        override fun onCreateView(
            inflater: LayoutInflater, 
            container: ViewGroup?,
            savedInstanceState: Bundle?
        ): View {
            _binding = FragmentFirstBinding.inflate(inflater, container, false)
            return binding.root
        }
        
        override fun onDestroyView() {
            super.onDestroyView()
            _binding = null
        }
    }
    ```
  </Step>
  
  <Step title="Navigation Graph">
    XML-based navigation graph defines all destinations and actions.
    
    ```xml
    <navigation
        android:id="@+id/nav_graph"
        app:startDestination="@id/FirstFragment">
        
        <fragment
            android:id="@+id/FirstFragment"
            android:name="com.example.vetycare.FirstFragment">
            <action
                android:id="@+id/action_FirstFragment_to_SecondFragment"
                app:destination="@id/SecondFragment" />
        </fragment>
    </navigation>
    ```
  </Step>
</Steps>

## View Binding Architecture

VetyCare uses **View Binding** for type-safe and null-safe view access. View binding is enabled in the app's `build.gradle.kts`:

```gradle
buildFeatures {
    viewBinding = true
}
```

### Benefits of View Binding

<AccordionGroup>
  <Accordion title="Type Safety">
    View binding generates a binding class for each XML layout file, providing compile-time safety for view IDs.
  </Accordion>
  
  <Accordion title="Null Safety">
    Eliminates `findViewById()` calls and potential `NullPointerException` errors.
  </Accordion>
  
  <Accordion title="Performance">
    Faster than reflection-based solutions like data binding or findViewById().
  </Accordion>
</AccordionGroup>

### View Binding Pattern in Fragments

All fragments follow this consistent pattern:

```kotlin
private var _binding: FragmentFirstBinding? = null
private val binding get() = _binding!!

override fun onCreateView(
    inflater: LayoutInflater, 
    container: ViewGroup?,
    savedInstanceState: Bundle?
): View {
    _binding = FragmentFirstBinding.inflate(inflater, container, false)
    return binding.root
}

override fun onDestroyView() {
    super.onDestroyView()
    _binding = null  // Prevent memory leaks
}
```

<Warning>
  Always set `_binding` to null in `onDestroyView()` to avoid memory leaks. The view hierarchy is destroyed when a fragment is placed on the back stack, but the fragment instance is retained.
</Warning>

## Android Jetpack Components

VetyCare leverages several Android Jetpack libraries for modern Android development:

| Component | Purpose | Version Reference |
|-----------|---------|------------------|
| **Core KTX** | Kotlin extensions for Android framework APIs | `libs.androidx.core.ktx` |
| **AppCompat** | Backward compatibility for newer Android features | `libs.androidx.appcompat` |
| **Navigation Component** | Fragment navigation and back stack management | `libs.androidx.navigation.fragment.ktx`<br/>`libs.androidx.navigation.ui.ktx` |
| **ConstraintLayout** | Flexible layout positioning and sizing | `libs.androidx.constraintlayout` |

### Dependencies Configuration

From `app/build.gradle.kts`:

```gradle
dependencies {
    implementation(libs.androidx.core.ktx)
    implementation(libs.androidx.appcompat)
    implementation(libs.material)
    implementation(libs.androidx.constraintlayout)
    implementation(libs.androidx.navigation.fragment.ktx)
    implementation(libs.androidx.navigation.ui.ktx)
    
    testImplementation(libs.junit)
    androidTestImplementation(libs.androidx.junit)
    androidTestImplementation(libs.androidx.espresso.core)
}
```

<Tip>
  VetyCare uses Gradle version catalogs (`libs.*` references) for centralized dependency management. This approach ensures consistent versions across modules.
</Tip>

## Architecture Benefits

This architecture provides several advantages:

1. **Simplified Navigation**: Navigation Component handles deep linking, safe args, and back stack management automatically
2. **Memory Efficient**: Single Activity reduces memory overhead compared to multiple activities
3. **Shared UI Elements**: Toolbar and FAB persist across fragment transitions
4. **Type Safety**: View binding and Kotlin eliminate entire classes of runtime errors
5. **Testability**: Fragment-based architecture enables easier unit and integration testing
6. **Material Design**: Built-in support for Material 3 components and theming
